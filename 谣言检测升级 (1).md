# 谣言检测升级
你们原来的分工是：

- **A**：用户输入 + Claim Extraction  
- **B**：Evidence Retrieval（RAG）  
- **C**：Fact Verification  
- **D**：Critic & Explanation  
- **E**：前端 & 系统集成  

在这个基础上，新的五人分工应该遵循一个原则：

> **每个人保留原模块主责，再各自往“URL输入、多模态、作者信息、联合判定”扩一层。**

这样最稳，不会推翻你们已经做好的东西。

# 二、每个人的详细工作

---

## A：输入解析负责人  
### 1. 保留原有工作
A 继续负责你们原来的两块：

- 文本输入
- Claim Extraction

也就是你们原来“输入一句文本，抽取可核查 claim”这条链还是 A 的核心任务。

### 2. 新增后的职责
A 现在要从“文本输入负责人”升级成 **统一输入入口负责人**。

新增负责：

- **URL 输入**
- **帖子内容解析**
- **作者基础信息抽取**
- **媒体资源抽取**
- **为后续模块输出统一结构化 PostObject**

### 3. A 要做的详细工作
#### 第一部分：扩展输入类型
把原来只有文本输入，改成支持：

- 文本输入
- URL 输入
- 图片上传
- 可选视频链接

至少第一版要支持：

- 普通网页 URL
- 社交媒体帖子 URL
- 图片型帖子

#### 第二部分：做 URL Resolver
输入一个 URL 后，先解析并标准化为统一对象，例如：

```python
{
  "platform": "...",
  "url": "...",
  "text": "...",
  "author": {...},
  "media": [...],
  "metadata": {...}
}
```

A 负责把不同来源统一成这一种格式，方便后面所有模块直接用。

#### 第三部分：升级 Claim Extraction
原来只对文本抽 claim，现在要变成对“扩展文本”抽 claim：

- 帖子正文
- 图片 OCR 文本
- 图片描述文本
- 视频转写文本（如果后期做视频）

并且最好输出 claim 来源：

```python
[
  {"claim": "...", "source": "post_text"},
  {"claim": "...", "source": "ocr"},
  {"claim": "...", "source": "caption"}
]
```

#### 第四部分：和 B/C/D 的接口
A 最终要输出两个对象：

1. `post_object`
2. `claims`

供 B 做检索，供 C 做判定，供 D 做解释。

### 4. A 的最终交付
A 最后应该交付：

- URL 输入解析模块
- 统一数据结构 `PostObject`
- 升级版 Claim Extraction
- 输入样例集（文本、网页、社媒、图片）

---

## B：检索负责人  
### 1. 保留原有工作
B 继续负责原来的：

- 文档库构建
- BM25 / 向量检索
- top-k 证据召回

这部分不变，仍然是系统事实证据的核心。

### 2. 新增后的职责
B 现在要从“单一事实检索”升级成 **双通道检索负责人**。

新增负责：

- **作者相关信息检索**
- **作者历史帖子抓取/整理**
- **权威来源优先检索**
- **多路 evidence 合并**

### 3. B 要做的详细工作
#### 第一部分：保留原事实证据检索
继续做：

- 新闻/百科/fact-check 数据源
- BM25 + embedding 双通道召回
- 对每个 claim 返回 top-k evidence

#### 第二部分：新增作者历史帖子检索
B 现在还要支持：

- 根据作者 ID 拉作者历史帖子
- 选择最近 N 条帖子
- 去重、清洗、结构化
- 返回给 C 和 D 做历史真实性分析

也就是说，B 不再只检索“这个 claim 的外部证据”，还要检索“这个发帖人过去发过什么”。

#### 第三部分：权威来源分级
B 要给 evidence 增加来源标签，例如：

- 官方机构
- 主流媒体
- fact-check 平台
- 普通网页
- 社交媒体二次传播

这样后面 D 才能做来源可信度分析。

#### 第四部分：多通道 evidence 输出
B 最好把 evidence 分成两类输出：

```python
{
  "claim_evidence": [...],
  "author_history_posts": [...],
  "source_profile": [...]
}
```

这样 C 做判定时不会把不同证据混在一起。

### 4. B 的最终交付
B 最后应该交付：

- 升级版检索器（claim evidence + author history）
- 权威来源标签系统
- top-k evidence 接口
- 作者历史帖子接口

---

## C：判定负责人  
### 1. 保留原有工作
C 继续负责原来的：

- claim + evidence → verdict
- SUPPORT / REFUTE / NEI
- 置信度输出

这部分仍然是整个系统最终“真假判断”的核心。

### 2. 新增后的职责
C 现在要从“文本事实核验”升级成 **联合判定负责人**。

新增负责：

- **多模态一致性判定**
- **作者特征接入**
- **历史真实性特征接入**
- **最终联合谣言分类**

### 3. C 要做的详细工作
#### 第一部分：保留原文本核验
继续做原来的：

- claim 与 evidence 是否一致
- 输出支持/反驳/证据不足
- 给出文本核验分数

例如：

```python
text_verdict = {
  "label": "REFUTE",
  "score": 0.86
}
```

#### 第二部分：新增多模态判定
C 要接收 A 传来的：

- OCR 文本
- 图片描述
- 图文一致性信息

然后判断：

- 图片和帖子文案是否冲突
- OCR 信息是否反驳原帖
- 视频文字/语音/画面是否一致（若后续扩展）

形成一个 `multimodal_score`。

#### 第三部分：新增联合分类器
C 要把这些分数融合起来：

- 文本核验分数
- 多模态分数
- 作者权威性分数
- 历史真实性分数

最后输出：

```python
{
  "rumor_prob": 0.87,
  "final_label": "高风险谣言",
  "subscores": {
      "text_score": ...,
      "mm_score": ...,
      "author_score": ...,
      "history_score": ...
  }
}
```

#### 第四部分：模型建议
C 最好不要只靠 prompt 直接拍结论，建议做：

- 第一层：LLM 做 claim verification
- 第二层：轻量融合模型做最终分类

比如逻辑回归、XGBoost、小 MLP 都可以。

### 4. C 的最终交付
C 最后应该交付：

- 文本核验模块升级版
- 多模态一致性判定模块
- 联合分类器
- 最终输出格式定义

---

## D：可信度与解释负责人  
### 1. 保留原有工作
D 继续负责原来的：

- 证据来源可信度分析
- explanation 生成
- 前端展示解释文本

这是你们系统“可解释性”的核心。

### 2. 新增后的职责
D 现在要从“证据解释员”升级成 **作者可信度建模负责人**。

新增负责：

- **作者基础可信度特征**
- **作者权威性评分**
- **作者历史真实性统计**
- **最终解释模板升级**

### 3. D 要做的详细工作
#### 第一部分：作者基础画像
D 要对作者做基础画像：

- 粉丝数
- 关注数
- 是否认证
- 账号年龄
- bio 简介
- 发帖频率
- 是否为机构号

注意：D 要把这些都定义为“弱特征”，不能直接当真值。

#### 第二部分：作者权威性建模
D 要做一个 `authority_score`，例如综合：

- 是否认证
- 是否机构媒体/政府/学校
- 发文领域是否与当前主题匹配
- bio 是否显示专业背景
- 历史发帖主题是否稳定

这样“粉丝多”就只是其中一个小项，而不是决定性指标。

#### 第三部分：历史真实性统计
这部分很关键。  
D 要根据 B 拉回来的历史帖子，统计：

- 最近 N 条帖子中，多少被系统判成 SUPPORT
- 多少被判成 REFUTE
- 多少是 NEI
- 是否常引用来源
- 是否常发布高风险话题

输出一个历史真实性概况，例如：

```python
{
  "support_rate": 0.52,
  "refute_rate": 0.28,
  "nei_rate": 0.20,
  "history_truth_score": 0.41
}
```

#### 第四部分：解释模块升级
D 最终解释不能只说“这个新闻是假”，而要解释成四层：

1. 文本与外部证据是否冲突  
2. 图像/视频与文案是否冲突  
3. 作者是否属于该领域权威来源  
4. 作者历史帖子真实性表现如何  

最后输出让老师一看就知道你们是“联合判定”，不是只做文本分类。

### 4. D 的最终交付
D 最后应该交付：

- author_profile 模块
- authority_score 计算规则
- history_truth_score 统计规则
- 升级版 explanation 模板

- 系统联调

### 2. 新增后的职责
E 现在要从“普通集成”升级成 **全链路产品化负责人**。

新增负责：

- **URL 输入框、图片上传区**
- **多分数可视化展示**
- **作者画像展示**
- **全模块接口联调**

### 3. E 要做的详细工作
#### 第一部分：输入界面升级
前端要从原来的单文本输入改成：

- 文本输入框
- URL 输入框
- 图片上传
- 可选平台识别提示

#### 第二部分：结果展示升级
结果页面建议拆成 5 块：

1. **原帖信息区**  
   - 原始文本  
   - URL  
   - 作者名称/认证/粉丝数  

2. **Claim 区**  
   - 抽取出的 claim 列表  
   - 每个 claim 来源  

3. **证据区**  
   - 检索到的外部证据  
   - 来源网站  
   - 摘要  

4. **作者画像区**  
   - 粉丝、认证、账号年龄  
   - authority_score  
   - 历史真实性概况  

5. **最终判定区**  
   - rumor_prob  
   - final label  
   - explanation  

#### 第三部分：系统集成
E 负责定义每个模块的输入输出接口，确保：

- A 输出能被 B/C/D 接住
- B 输出能给 C 和 D
- C 输出能给 E 展示
- D 的 explanation 能无缝挂到最终结果页

#### 第四部分：演示逻辑
E 最好提前准备 2 到 3 条可展示样例：

- 纯文本谣言
- 图文不一致谣言

这样答辩会更完整。

- 演示样例与展示流程
- 最终可运行 demo

# 三、最合理的协作关系

为了避免五个人各做各的，建议你们按下面的链路协作：

### A → B
A 输出：

- `post_object`
- `claims`

B 用来做检索。

### B → C
B 输出：

- `claim_evidence`
- `author_history_posts`

C 用来做联合判定。

### B → D
B 输出：

- `source_profile`
- `author_history_posts`

D 用来算来源可信度和作者历史真实性。

### C + D → E
C 输出最终分数，D 输出解释文本，E 统一可视化。
